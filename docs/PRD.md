# Cofree PRD（基于 v0.0.7 当前实现）

## 1. 产品定位

Cofree 是一个 **Tauri 2 桌面端 AI 编程助手**。它不是独立部署的 Web 服务，而是运行在用户本机、面向单个工作区工作的桌面应用。

当前产品形态可以概括为：

- 用户在桌面应用中选择本地工作区并配置模型连接。
- 聊天页负责提出任务、查看流式回复、审阅待执行动作。
- 编排层通过工具调用循环读取代码、生成 patch、调用 shell、拉取诊断、委派子 Agent。
- 涉及写盘或命令执行时，系统通过审批门暂停，等待用户确认。

## 2. 目标用户

当前更适合以下使用方式：

- 独立开发者或小团队成员，在本地仓库上完成修改、验证与提交准备。
- 希望保留人工审批，而不是把写文件/执行命令完全交给模型的人。
- 已经具备自己的模型 API Key 或本地模型/网关地址的用户。

## 3. 当前核心页面

### 3.1 聊天页
- 输入需求并查看流式回复。
- 展示 `<think>` 折叠内容、工具调用状态、错误提示。
- 承载待审批 patch、shell 动作、批量审批和结果回显。

### 3.2 设置页
- 管理工作区路径、模型供应商、模型列表、代理和工具权限。
- 选择内置 ChatAgent 或维护自定义 Agent。
- 控制是否允许云模型、代码片段长度、上下文上限、路径发送策略。

### 3.3 控制台页
- 查看工作流阶段、请求统计、工具调用追踪。
- 浏览 LLM 请求与敏感动作审计日志。
- 导出 JSON / CSV 审计记录。

## 4. 当前产品主流程

1. 用户启动应用并选择一个本地目录作为工作区。
2. 用户在聊天页发出需求。
3. 编排层根据上下文决定调用 `list_files`、`read_file`、`grep`、`glob`、`git_status`、`git_diff`、`diagnostics`、`fetch` 等工具。
4. 如有必要，主 Agent 可通过 `task` 把子任务委派给 `planner`、`coder`、`tester` 子 Agent。
5. 需要写盘时，系统生成 patch 并展示 diff 审批卡片。
6. 需要执行命令时，系统展示 shell 审批卡片。
7. 审批通过后，系统执行动作、记录审计、更新会话状态，并在适用时运行自动诊断。

## 5. 当前已实现的主要能力

### 5.1 代码理解与上下文收集
- 工作区文件读取、目录枚举、内容搜索、glob 搜索。
- Git 状态与工作区 diff 查询。
- 项目级 `.cofreerc` 读取，支持 `systemPrompt`、`ignorePatterns`、`toolPermissions`、`contextFiles`、`language`、`overviewBudget`。

### 5.2 编排与 Agent 能力
- 顶层 ChatAgent：全栈工程师、代码审查员、架构师、QA 工程师。
- 内部 Sub-Agent：`planner`、`coder`、`tester`。
- 原生工具调用循环、长对话压缩、多交付物检测、工具轮次限制。

### 5.3 修改与执行
- `propose_file_edit` 与 `propose_apply_patch` 两种 patch 提交流程。
- diff 预检、文件快照、失败回滚、自动诊断。
- `propose_shell` 执行 shell 命令，支持超时控制与结果展示。

### 5.4 可观察性与恢复
- LLM 请求与敏感动作审计日志本地持久化。
- SQLite checkpoint 恢复工作流状态。
- 控制台页可查看工具时间线、统计信息、日志导出。

### 5.5 模型与连接管理
- 多供应商、多协议模型配置。
- 支持 `openai-chat-completions`、`openai-responses`、`anthropic-messages`。
- 可手动维护模型、拉取模型列表、为 Agent 绑定固定模型。
- 支持代理配置和“允许云模型”开关。

## 6. 当前产品边界

以下内容不应在文档中被描述为“已经具备”：

- 强隔离 sandbox。
- 独立 Git 提交向导或专用 commit UI。
- 复杂 Git 场景的自动处理，例如 submodule、LFS、merge/rebase 冲突修复。
- 并行 Sub-Agent 编排。
- 基于未来里程碑的承诺。

## 7. 文档使用原则

本 PRD 只对齐当前实现。若某项能力尚未能从仓库代码或配置中确认，就不写入主文档。未来想法应独立记录，不回填到这里。
