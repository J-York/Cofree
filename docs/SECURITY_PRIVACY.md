# Cofree Security & Privacy（v0.0.9）

本文件只描述当前版本真实存在的安全与隐私机制。

## 1. 基本原则

1. Cofree 是本地桌面应用，文件操作、patch 应用、shell 执行、审计记录都发生在用户机器上。
2. 模型请求会把必要上下文通过 HTTP 发往用户配置的模型端点，因此并不是“绝对离线”。
3. 对文件写入和命令执行的控制主要依赖审批门、工具权限、路径校验和命令拦截，而不是强隔离沙箱。

## 2. 数据外发边界

### 2.1 可能外发的数据
- 用户在聊天中输入的自然语言需求。
- 为完成任务而选取的代码片段、错误输出、文件路径信息。
- 工具调用产生的结构化上下文。
- 手动使用 `fetch` 获取的远程 URL 内容。

### 2.2 当前已有的控制项
- `allowCloudModels`：控制是否允许云模型。
- `maxSnippetLines`：限制代码片段长度。
- `maxContextTokens`：限制上下文预算。
- `sendRelativePathOnly`：默认只发送相对路径。
- `toolPermissions`：可限制写工具与 `fetch` 的执行方式。

### 2.3 当前没有实现的限制
- `fetch` 没有内置域名白名单。
- 没有针对所有模型请求的全局 DLP 扫描器。
- 没有操作系统级别的强隔离网络沙箱。

## 3. API Key 存储

当前 API Key 的存储方式是：

- 通过 Rust 端的加密文件存储到 `~/.cofree/`。
- 主密钥写入 `~/.cofree/keystore.key`。
- 加密后的条目写入 `~/.cofree/keystore.json`。
- Unix 上文件权限以 `0600` 创建。
- 运行时会有内存缓存。
- 如检测到旧 keyring 条目，会迁移到新存储后再删除旧条目。

因此，当前实现 **不是** 仅依赖 macOS Keychain，也不是把 API Key 放进 `localStorage`。

## 4. 本地持久化数据

### 4.1 `localStorage`
- 应用设置（不含明文 API Key）。
- 会话列表与消息元数据。
- LLM 请求审计日志。
- 敏感动作审计日志。

### 4.2 `~/.cofree/`
- `checkpoints.db`：工作流恢复数据。
- `snapshots/`：patch 失败回滚快照。
- `keystore.key` / `keystore.json`：加密凭据存储。

## 5. 审计日志

### 5.1 记录内容
- LLM 请求：provider、model、requestId、输入长度、输出长度、时间等元数据。
- 敏感动作：actionId、类型、状态、开始/结束时间、执行者、原因、工作区路径、详情。

### 5.2 当前特性
- 每类日志最多保留 200 条。
- 写入前会做脱敏和持久化清洗。
- 可在控制台页导出 JSON / CSV。

## 6. 写操作安全模型

### 6.1 Patch
- 默认需要审批。
- 应用前先做预检。
- 执行前创建快照。
- 执行失败时自动回滚。

### 6.2 Shell
- 默认需要审批。
- 用户看到完整命令文本后再决定是否执行。
- Rust 后端会拦截一组灾难性命令模式。

## 7. 路径与工作区边界

- 后端拒绝绝对路径。
- 后端拒绝包含 `..` 的路径穿越。
- 目标路径必须留在工作区内。
- 工作区本身只要求是存在的目录；不是强制必须为 Git 仓库。

## 8. 风险说明

以下风险是当前产品模型的一部分，文档不应弱化：

- 一旦用户将写工具配置成 `auto`，写盘或命令执行就可能跳过人工审批。
- 如果用户连接云模型，代码与输出会按实际请求内容离开本机。
- 如果用户批准 shell 命令，系统会在工作区中真实执行该命令。
- `fetch` 可以访问任意可达 URL，是否使用取决于工具权限和具体任务。
